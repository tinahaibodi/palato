# Palato Onboarding Redesign — Design Document

**Date:** 2026-03-02
**Status:** Approved

---

## Summary

Palato shifts from a CLI tool that scaffolds a blank template to a **skill-first system** where the agent itself conducts a taste interview, scrapes external references, and synthesizes a filled profile. The CLI shrinks to a thin installer. The MCP server is removed.

---

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Execution model | Skill-first | The agent is the runtime. Interview is conversational — that's what skills do. |
| External content | URL-fetch-and-summarize | No API auth. Agent fetches public URLs, extracts taste signals. 80% of value, 20% of complexity. |
| Profile naming | Profile-per-context | One user may have multiple profiles (e.g. brutalist, warm). Switchable via pointer. |
| Profile storage | `palato/` at project root | Single source of truth. Skills folder gets a symlink. |
| Profile loading | On-demand | Agent reads the active profile only when making design/aesthetic decisions, not on every conversation. |
| Profile tone | Tight synthesis | Agent distills answers into concise design principles. Reads like a creative brief, not a transcript. |

---

## Architecture

### File Structure

```
palato/
├── palato.md               ← Skill instructions + active profile pointer
├── keeks-palato.md         ← Profile
├── brutalist-palato.md     ← Profile
└── links.md                ← User-curated reference URLs (optional)

.claude/skills/palato.md    → symlink to palato/palato.md
```

- `palato/palato.md` is the skill file. It contains interview instructions, synthesis logic, runtime behavior, and a pointer to the active profile.
- Generated profiles live alongside it in `palato/`.
- The agent's skills folder gets a symlink so the agent loads the skill instructions automatically.
- Only one profile is "active" at a time. The pointer in `palato.md` determines which one.

### Active Profile Pointer

At the top of `palato/palato.md`:

```markdown
<!-- active: keeks-palato.md -->
```

Agent reads this to know which profile to load. Updates it on switch.

## Onboarding Flow

### Trigger

User says `/palato [name]` (or equivalent). Agent checks if `palato/[name]-palato.md` exists.

- If **yes** → load it, set as active. Done.
- If **no** → start interview.

### Interview Sequence

**Step 1 — Core Questions (mandatory)**

> **Q1 — Where did you come from?**
> *(childhood, culture, where you grew up, how you were raised, hardship, spirituality, inner life)*

> **Q2 — What made you?**
> *(artists, directors, writers you've obsessed over, your training, the era that formed you, failure, love, work you wish you'd made)*

One at a time. Wait for answer before proceeding.

**Step 2 — Design Questions Gate**

> Want to answer a few quick design questions too? (~2 minutes, optional)

If **yes** → five questions, one at a time:

| # | Topic | Prompt |
|---|-------|--------|
| Q3 | Components & UI | Describe a UI component you love and one you hate. Name the product, what it looks like, why it feels right or wrong. |
| Q4 | Typography | Name a typeface or style you'd use for headings. One you'd never touch. What does good type feel like? |
| Q5 | Colour | Describe your colour instincts — palette, mood, what you'd never put on screen. |
| Q6 | Motion | How should things move? Fast or slow, snappy or floaty, when should things animate at all? |
| Q7 | Layout & Spacing | Tight and dense or generous and airy? What does cluttered mean to you? |

If **no** → skip to Step 3.

**Step 3 — External Links Gate**

> Want to link to some sites you've curated?

Options: Paper, Figma, Are.na, or other URLs. Agent fetches each publicly accessible URL, summarizes the content, extracts taste signals.

If declined → skip.

**Step 4 — links.md Gate**

> Want to add more reference links? (create a `links.md` in `palato/` and I'll read it)

If yes → agent creates `palato/links.md` with a template. User populates it. Agent reads and processes.

If no → skip.

**Step 5 — Synthesis**

Agent takes all inputs (answers + scraped content) and generates `palato/[name]-palato.md`. Updates the active pointer in `palato/palato.md`.

## Profile Output Format

### Minimal Profile (Q1+Q2 only)

```markdown
# [name] — Taste Profile
> Generated by Palato. Ground truth for design, copy, and aesthetic decisions.

## Origin
<!-- Synthesized from Q1 — distilled taste signals from background -->

## Influences
<!-- Synthesized from Q2 — taste principles from artistic/cultural references -->

## Inferred Aesthetic
<!-- Agent's synthesis of visual/tonal preferences from Q1+Q2. Marked as inferred. -->

## Anti-Patterns
<!-- What to avoid — derived from stated aversions and inverse of preferences -->
```

### Full Profile (Q1+Q2 + design questions)

All of the above, plus:

```markdown
## Visual Language

### Components & UI
### Typography
### Colour
### Motion
### Layout & Spacing
```

### With External Links

Scraped content enriches all relevant sections. Taste signals from an Are.na board of brutalist typography flow into the Typography section — not dumped in a separate block.

## Skill File Modes

The skill file (`palato/palato.md`) operates in three modes:

### Mode 1 — Init

Triggered when `/palato [name]` is called and no matching profile exists. Runs the interview sequence described above. Contains the exact question phrasing verbatim — agent must present them as-written, not paraphrase.

### Mode 2 — Runtime

Triggered during normal work when the agent makes design, UI, copy, or aesthetic decisions. Instructions:

- Read `palato/[active profile]` before making visual, tonal, or architectural decisions.
- Apply as constraints, not suggestions.
- If a decision conflicts with the profile, flag it — don't silently override.

### Mode 3 — Switch

Triggered when user says `/palato use [name]` or "switch to the brutalist profile."

- Update the active pointer in the skill file.
- Confirm the switch.
- If the named profile doesn't exist, offer to create it (start init flow).

## CLI Installer

```
npx palato install
```

**What it does:**

1. Creates `palato/` directory at project root.
2. Copies the skill file template into `palato/palato.md`.
3. Detects installed agents (Claude Code, Cursor, Codex, OpenCode).
4. Creates the appropriate symlink or config injection per agent:

| Agent | Wiring |
|-------|--------|
| Claude Code | Symlink `.claude/skills/palato.md` → `palato/palato.md` |
| Cursor | Symlink in `.cursor/rules/` or injection in `.cursorrules` |
| Codex | Reference injected into `AGENTS.md` |
| OpenCode | Equivalent config injection |

**What it no longer does:**

- No interactive Q&A (the agent handles this now)
- No MCP server wiring (no MCP server)
- No `doctor` command
- No `server` command

**Dependencies removed:** `@modelcontextprotocol/sdk`, `zod`, `inquirer`, `ora`
**Dependencies kept:** `commander`, `chalk`, `fs-extra`

---

## What Gets Deleted

The following are no longer needed:

- `src/mcp/server.js` — MCP server
- `src/mcp/tools.js` — MCP tool handlers
- `src/cli/doctor.js` — Doctor command
- Most of `src/cli/init.js` — Interactive wizard (replaced by skill-driven interview)
- `src/templates/PALATO.md` — Blank template (replaced by agent-generated profiles)

---

## Agent Compatibility

The skill file is agent-agnostic markdown. Any agent that supports instruction files and basic file operations (read, write, web fetch) can run it. The only agent-specific part is the symlink/wiring location, handled by the CLI installer.

Fallback for unsupported agents: user manually tells the agent "read `palato/palato.md` and follow its instructions."
